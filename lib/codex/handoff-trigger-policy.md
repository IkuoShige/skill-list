# Handoffトリガーポリシー（客観指標）

## 5つの客観指標

| 指標 | 閾値 | 計測方法 |
|------|------|----------|
| コンテキスト使用率 | 60%+ | CCの自己推定（会話長・ツール出力量・ファイル読み込み量から判断） |
| ターン数 | 30+ | 会話のラウンドトリップ数（概算） |
| ツール呼出回数 | 50+ | セッション中のツール使用回数（概算） |
| diff行数 | 500+ | `git diff --stat` の合計変更行数 |
| セッション時間 | 2時間+ | セッション開始からの経過時間（概算） |

## トリガー条件

### 通常モード（手動セッション）
- 5指標のうち **2つ以上** に該当した場合、handoffを提案する
- 提案のみ。強制しない。ユーザーが却下したら同一セッション内で再提案しない

### /auto モード（自律ループ）
- **コンテキスト使用率 60%+** 単独でトリガー（他指標を待たない）
- 即座にセッション引き継ぎ＆自動再開を実行する
- コンテキスト圧縮が始まる前に引き継ぐことが最優先
- **Training実行中の場合:**
  - コンテキスト使用率チェックは監視ループ内でも継続（10イテレーションごと）
  - Handoff時はTrainingプロセスを **killしない**（バックグラウンドで継続させる）
  - 引き継ぎ文書にTraining状態（PID, run_id, logfile, 最新メトリクス）を記録する

## コンテキスト使用率の推定方法
CCは自身のコンテキスト消費を正確に計測できないが、以下のヒューリスティクスで推定する:
- 会話のターン数が多い（20+ターン → 概算40%以上）
- 大きなファイルを複数回Read した
- ツール出力が大量（長いdiff、長いログ）
- システムから「context compression」の通知が来た → **即座にhandoff（推定100%近い）**

## 提案メッセージ
```
セッションが長くなっています（{該当指標を列挙}）。
引き継ぎ文書を作成しますか？ `/handoff` で実行できます。
```

## 注意事項
- 指標の計測は厳密である必要はない（概算で十分）
- diff行数はgit管理下のプロジェクトでのみ計測可能
- このポリシーはCLAUDE.mdのhandoff awarenessセクションから参照される
