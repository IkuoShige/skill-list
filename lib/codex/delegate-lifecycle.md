# delegate-codex ライフサイクル

## 4段階ライフサイクル

### 1. 委託（Dispatch）
- 委託プロンプトを構築し、Task fork（run_in_background: true）で起動する
- Codexは `--full-auto` で実行され、ファイル操作・コード変更を含む全操作が可能

### 2. CC継続（Continue）
- Codexがバックグラウンドで処理中、CCはユーザーとの対話を継続できる
- ユーザーに「Codexにバックグラウンドで委託しました。結果を確認するには "check codex" と言ってください」と通知する
- タスクIDを記録しておく

### 3. 結果確認（Check）
- ユーザーが「check codex」と言ったら、TaskOutputでバックグラウンドタスクの結果を取得する
- Codexの応答をJSONLから抽出する（codex-common.mdの解析ルールに従う）

### 4. 結果提示（Present）
- Codexの応答を整形してユーザーに提示する
- Codexがファイル変更した場合は git diff --stat で変更概要を提示する

## 委託プロンプト構築ガイド
- Codexはプロジェクトのファイルを直接読み書きできる
- 必要な情報を明示的にプロンプトに含める:
  - 関連コードのパスまたは内容
  - 設計上の制約・前提
  - 具体的な分析観点・質問
- 曖昧な「よろしく」系の委託は禁止。具体的なアウトプットを指定する

## デフォルト委託プロンプトテンプレート

プロジェクトローカル（`{project}/.claude/prompts/delegate-prompt.md`）が優先。
なければ以下をデフォルトとして使用する:

```
以下のタスクに取り組んでください。ファイルの読み書き・コード変更を含め、必要な操作はすべて実行してよい。

## タスク
{task_description}

## プロジェクト文脈
- リポジトリ: {repo_path}
- 関連ファイル: {relevant_files}
{additional_context}

## 期待するアウトプット
{expected_output}

## 制約
- 具体的な提案を出すこと（抽象論は不可）
- トレードオフがある場合は明示すること
- 結論に至る推論過程を示すこと
```

変数:
- `{task_description}`: ユーザーの依頼内容
- `{repo_path}`: CCの現在の作業ディレクトリ（pwdで取得）
- `{relevant_files}`: CCが収集した関連ファイルのパス一覧
- `{additional_context}`: CCが収集したコード片・設計文脈（任意）
- `{expected_output}`: 具体的な成果物の形式

## 制約事項
- 自動ポーリングなし: ユーザーが明示的に確認を求めるまで結果を取りに行かない
- 1委託ずつ: 同時に複数のCodex委託を実行しない（結果管理の複雑化を避ける）
